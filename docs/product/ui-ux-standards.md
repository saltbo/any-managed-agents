# UI/UX Standards

Any Managed Agents is an operational control plane. The product UI must feel quiet, dense, predictable, and built for repeated work. New pages must reuse the same application shell, component primitives, route patterns, spacing, and feedback model instead of inventing local page styles.

## Product Style

- Use a work-focused SaaS/admin style: restrained surfaces, high information density, clear resource lists, predictable actions.
- Do not use marketing-page layout patterns inside the console: no hero sections, decorative gradients, oversized cards, or explanatory feature blocks.
- Use neutral semantic tokens from the shadcn theme. Components must not introduce raw hex colors or one-off palette systems.
- Use lucide icons for navigation and icon-only actions. Icon-only buttons must have an accessible label.
- Each page has one primary action in the top command area. Secondary and destructive actions are visually subordinate.

## Architecture

- `src/App.tsx` may only compose application providers and `RouterProvider`.
- `src/app/router.tsx` owns primary UI routes.
- `src/features/<feature>/` owns route pages and feature orchestration.
- Shared console shell, resource context, and global command behavior live under `src/features/console/`.
- Shared context is not a page-operation service locator. It may expose authenticated resource state, filters, selection, refresh, and global create-sheet entry points. Feature operations such as archive, stop, disconnect, and page-specific mutations must live in the owning `src/features/<feature>/` module.
- Reusable business UI such as resource cards, empty states, form shells, badges, and metadata rows lives under `src/console/`.
- Generic primitives must come from `src/components/ui/` generated by shadcn. Product pages must compose these primitives instead of direct ad hoc HTML controls.
- Server state must use `@tanstack/react-query` queries and mutations. Do not add feature-level `useEffect + useState` API loading loops.
- New route modules must be deep-linkable and should not be controlled only by in-memory view state.

## Shell And Navigation

- The console shell is shared across all authenticated pages.
- Desktop uses the persistent left sidebar and top command area.
- The right content area must not add a duplicate sticky topbar. Page-level headers belong inside route content.
- The authenticated user entry is part of the left shell: desktop places it in the sidebar footer, and mobile may pin it to the lower-left. It opens a dropdown menu. On desktop the menu opens from the right side of the sidebar. Logout is a direct menu action and must not use a submenu arrow.
- Mobile uses horizontal primary navigation with full labels: `Agents`, `Environments`, `Sessions`, and later feature names. Labels must not be ellipsized.
- Primary route labels, page headings, and browser URLs must agree.
- Navigation must be links for route changes, not buttons that only mutate local state.
- Disabled or unavailable sections may appear in navigation only when visually disabled and non-interactive.

## Layout

- Use `max-w-6xl` as the default content width for console pages.
- Use the 4px/8px spacing scale already expressed by Tailwind/shadcn classes.
- Resource browsing is the primary first viewport. Creation, debug, or acceptance-test content must not occupy the default resource browsing surface.
- Cards are allowed for individual resource records, empty states, modals, and sheets. Do not put cards inside cards.
- Prefer two-column metadata grids on desktop and single-column flow on mobile.
- Mobile layouts must avoid horizontal page scroll and must keep primary actions reachable without requiring precision taps.

## Components

- Use shadcn `Button`, `Card`, `Badge`, `Input`, `Textarea`, `Select`, `Checkbox`, `Sheet`, `Alert`, `AlertDialog`, `Field`, `ScrollArea`, `Separator`, `Tabs`, and `Skeleton` before adding custom primitives.
- Product components may wrap shadcn primitives to encode AMA-specific patterns such as `StatusBadge`, `Meta`, `EmptyState`, and resource cards.
- Do not create new global utility classes like `.panel`, `.button-primary`, `.icon-button`, `.field-label`, `.input`, or `.textarea`.
- Do not style native buttons/inputs/selects directly in route pages when an existing shadcn primitive exists.
- Card radius, button radius, focus rings, disabled states, and density must come from the shared shadcn theme.
- Resource date and time display must use the shared `formatDate` helper backed by `dayjs`; route pages must not instantiate local date formatting.

## Forms

- Creation and edit flows open in a `Sheet` or a dedicated route. They must not be permanently visible side panels on resource list pages.
- Every field must have a visible label. Placeholder-only labels are forbidden.
- Forms must use shadcn `FieldGroup`, `Field`, `FieldLabel`, `FieldDescription`, and `FieldError` instead of local label wrappers or ad hoc field grids.
- Complex fields must include persistent helper copy when the expected format is not obvious.
- Form submit buttons must show pending/disabled state when mutation is in flight.
- Field-specific validation belongs near the field. Blocking page-level failures may use the shared `Alert` pattern in the affected surface.
- Destructive actions such as archive, delete, and stop must use the shared confirmation dialog before executing.

## Feedback And State

- Loading states use skeletons or scoped pending controls for resource areas. Full-page loading is only for initial auth/project resolution.
- Empty states must describe what is missing and, when allowed, expose the primary creation action.
- Runtime/session errors must be visible in the session detail surface and must not be hidden in raw JSON.
- Completed operations such as create, save, archive, stop, disconnect, copy, and download must use the shared toast component. They must not render success or failure text inside the page content flow.
- Runtime/session errors remain in the transcript or debug surface because they are inspectable session data, not transient operation feedback.
- Long-running actions such as session start and task send must disable repeated submits while pending.

## Sessions UX

- Session detail is a work surface, not a raw log dump.
- Transcript content is primary; debug/runtime event data is secondary and should move behind tabs or a collapsible debug area as the UI matures.
- Task input stays close to the selected session detail and uses the shared form controls.
- Session list items must make status, agent identity, and session id scannable without relying on color alone.

## Accessibility

- All interactive elements must be reachable by keyboard and have visible focus states.
- Icon-only actions require `aria-label`.
- Use semantic links for navigation and semantic buttons for commands.
- Maintain a valid heading hierarchy: shell page title is `h1`, resource cards use lower-level headings.
- Normal text must meet WCAG AA contrast. Do not introduce gray-on-gray local styling.
- Do not communicate state by color alone; include labels, icons, or text.

## Responsive Rules

- Mobile-first constraints apply at 390px width.
- No mobile route may require horizontal page scroll.
- Primary touch targets must be at least 44px effective size where space allows; compact desktop controls may use shadcn small sizes.
- Long identifiers and model names must wrap or break safely inside their containers.
- Never use viewport-scaled font sizes. Use the shared text scale.

## Implementation Checklist

Before adding or changing a page:

1. Add or update the relevant `specs/product/*.feature` scenario.
2. Use a route in `src/app/router.tsx`.
3. Put route orchestration in `src/features/<feature>/`.
4. Keep feature mutations in the owning feature module; do not add page operations to shared console context.
5. Use React Query for server state.
6. Compose from shadcn primitives and shared AMA product components.
7. Verify with `npm run test:e2e`, `npm run test`, `npm run typecheck`, `npm run lint`, and `npm run build`.
8. For visible UI changes, verify desktop and 390px mobile screenshots with Playwright.
